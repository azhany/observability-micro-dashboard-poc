# Story 2.1: Base Dashboard Layout & Tenant View

## Status
Ready for Review

## Story
**As a** System Operator,
**I want** a modern, dark-mode dashboard interface,
**so that** I can view a high-level summary of my tenant's health.

## Acceptance Criteria
1. Vue 3 dashboard features a sidebar for navigation and a main content area for widgets.
2. An "Overview" page displays a list of active agents and their last reported status.
3. A "Tenant Detail" page exists that accepts a tenant ID and prepares placeholders for metric charts.
4. UI is styled using Tailwind CSS with a consistent "Technical/NOC" aesthetic.

## QA Results

### Review Date: 2026-01-18

### Reviewed By: Quinn (Test Architect)

### Code Quality Assessment
The implementation successfully delivers the "NOC-First" dark mode aesthetic and basic navigation. The Vue components are well-structured and use the custom Tailwind theme correctly. The use of Lucide icons matches the technical specification.

However, there are technical concerns regarding performance and security as the system scales beyond a PoC:
1. **Performance (PERF-001)**: The dashboard overview performs a `DISTINCT` query on the `metrics_raw` table. On large datasets, this will cause significant latency.
2. **Security (SEC-001)**: Tenant isolation is not yet enforced at the controller level.
3. **Task Completion Mismatch**: Task 2 was marked as complete, but `auth` middleware has not been applied to the routes in `routes/web.php`.

### Refactoring Performed
- **File**: `app/app/Http/Controllers/DashboardController.php`
  - **Change**: Added explanatory comments for PoC shortcuts and security considerations.
  - **Why**: To document technical debt and ensure future developers are aware of the scaling limitations.

### Security Review
**CONCERN**: Lack of `auth` middleware on dashboard routes and lack of tenant isolation checks in `DashboardController::show`. While acceptable for an early PoC, these must be addressed before any multi-tenant deployment.

### Performance Considerations
**CRITICAL**: The `SELECT DISTINCT agent_id` on `metrics_raw` is a scalability bottleneck. Recommendation is to maintain an `agents` metadata table or a materialized view.
